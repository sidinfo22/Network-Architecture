Section 2

Company Objectives & Gap Analysis

Summit Care Medical Clinic is framed as a New York City healthcare office that handles patient information, office operations, scheduling activity, billing processes, remote collaboration needs, and internal administrative workloads in a HIPAA-aware environment.

Current-State Conditions

The starting environment reflects a small office network built around a basic Verizon router, shared-password wireless access, manually configured desktops, limited endpoint uniformity, minimal centralized controls, and fragmented visibility into users, systems, and recovery readiness.

Target-State Direction

The target environment introduces Windows domain services, identity-based wireless access, stronger segmentation, hybrid Microsoft identity, standardized endpoints, recovery planning, role-based access, cleaner documentation, and a much more supportable operational baseline.

Primary Objectives

  • Replace flat small-office networking with segmented enterprise-style architecture.
  • Standardize user desktops and laptops for easier provisioning and scaling.
  • Introduce Active Directory, DNS, DHCP, NPS/RADIUS, and centralized Windows identity.
  • Replace shared wireless passwords with WPA2-Enterprise / identity-backed wireless access.
  • Shift from Google Workspace direction to Microsoft 365 and Entra-connected services.
  • Support secure remote work, remote support, VPN access, Teams collaboration, and future office growth.
  • Improve documentation, asset visibility, backup readiness, and HIPAA-aware administration.

Section 3

Research & Development

The research phase evaluated the stack before procurement and deployment began. This included hardware platforms, cloud ecosystems, virtualization options, software choices, licensing paths, VPN models, support tooling, and staff readiness. The point was not to collect abstract ideas, but to make practical choices that matched the clinic’s needs, budget expectations, and support realities.

Network Platform Research

Cisco and Ubiquiti were compared for routing, switching, controller management, compatibility, support depth, training overhead, and long-term scalability. Cisco offered stronger enterprise networking depth, while Ubiquiti remained useful in the wireless layer due to practical availability and compatibility with the switching plan.

Cloud & Productivity Research

Google Workspace and Microsoft were compared for identity, productivity, email, licensing, security features, and hybrid integration. Microsoft was selected because it aligned more naturally with the Windows domain model, Entra synchronization, Teams, OneDrive, and role-based cloud administration.

Virtualization Research

Proxmox, Hyper-V, and VMware were reviewed for cost, usability, familiarity, virtualization capabilities, and recovery considerations. Proxmox emerged as the practical fit because it was cost-effective, Linux-based, and aligned with existing technical skill sets.

Section 6

Infrastructure Deployment & Configurations

Deployment for Summit Care is based on golden baselines, repeatable build standards, and documented administration. The purpose is not only to stand the environment up once, but to make it supportable, auditable, and scalable.

Core Deployment Stack

  • Proxmox is installed on the Dell PowerEdge R740XD as the type-1 hypervisor.
  • Windows Server 2022 VM provides Active Directory, DNS, DHCP, and NPS/RADIUS.
  • Additional Linux VPN VM supports software-based backup remote access.
  • Test/sandbox VM supports validation before changes are pushed broadly.
  • iDRAC provides out-of-band recovery and management if host-level issues occur.

Endpoint Deployment Method

Workstations are provisioned from USB-based Windows 11 media using Rufus and the Windows ISO. After installation, endpoints are renamed according to the SCMC standard, joined to Active Directory, brought under Group Policy, and loaded with the approved software stack and OneDrive / Teams / Microsoft sign-in posture expected in the clinic.

DHCP Failover & Addressing Logic

DHCP is centered on Windows Server for internal address management because it aligns directly with the domain environment, reservations, DNS updates, and user support workflow. The Cisco router also retains DHCP capability as part of the continuity model. In practice, that means the Windows server handles the primary office scopes while the Cisco router remains configured to support fallback or scoped continuity if the Windows DHCP role becomes unavailable. Critical infrastructure devices such as the firewall, router, switches, NAS, printer, and iDRAC remain on documented static or reserved addressing rather than depending on dynamic leases.

The design intentionally keeps Windows DHCP as the normal source of truth because it integrates better with Active Directory, DNS behavior, reservations, and day-to-day support. The Cisco router is not presented as a replacement for the Windows role; it is a continuity measure that allows basic scoped address service to remain available if the Windows DHCP role is down. This helps reduce downtime risk while preserving static visibility for every critical management and infrastructure device.

Cisco Configuration Workflow

Cisco devices are administered over secured management paths using SSH rather than open, unauthenticated access. Private keys are created, stored securely, and referenced through the documented admin workflow. Running configurations are saved, exported, and retained in multiple protected locations so the team can rebuild or re-import configurations if hardware must be replaced. The documented workflow includes validating the running config, writing changes to startup config, exporting backup copies, and retaining those copies on the server, NAS, and cloud documentation locations.

VPN Configuration & Redundancy Model

Summit Care’s remote-access design uses two different VPN paths. The Cisco ASA serves as the primary edge VPN because it sits directly in the firewall path and can enforce the same perimeter restrictions used elsewhere in the office security posture. A Linux-based software VPN virtual machine is also deployed within the Proxmox environment as a secondary remote-access path. That internal software VPN provides administrative recovery flexibility and preserves remote support access if the primary firewall-side VPN path must be serviced, changed, or temporarily taken offline.

This dual-VPN model is important because it supports both resilience and control. Staff and IT can rely on the ASA as the normal business-facing path, while the Linux-based VPN remains documented and available for backup continuity. The two paths are not meant to create uncontrolled duplication; they exist so remote access remains recoverable without improvising insecure access during an outage or maintenance event.

Automation & Scheduled Operations

PowerShell scripts are stored on the server’s protected local volume and run through Task Scheduler to handle recurring support tasks such as backup routines, operational checks, log review triggers, and alert workflows. Linux-based support routines complement the Windows side where needed, while SNMP monitoring provides threshold-based visibility into device health and service stability.

Section 7

Hybrid Cloud Infrastructure & Scalability

Summit Care’s hybrid cloud approach is centered on Microsoft identity and service integration, not on moving every workload into the cloud. The local server environment remains the operational anchor while Entra and Microsoft 365 extend identity, password management, collaboration, licensing, and cloud-side user administration.

Hybrid Rollout Method

  1. Microsoft Entra Connect is installed on the domain controller.
  2. The local UPN / trust model is aligned to match the cloud identity format.
  3. Synchronization is established so users and changes flow between local AD and cloud identity.
  4. Password hash sync, password writeback, SSPR, and MFA are enabled as part of the hybrid identity posture.
  5. Microsoft 365 apps, Teams, OneDrive, and related cloud services are tied into the same sign-in workflow.

A full Azure VM stack was considered as a future growth option, especially in the context of another office or expanded hybrid operations, but the current design intentionally stops short of that and focuses on hybrid identity, cloud app access, sync monitoring, and scalable Microsoft administration without overbuilding the first phase.

Network Segmentation, VLANs & Subnetting

Segmentation is one of the main architectural controls that separates the upgraded Summit Care environment from a simpler flat office network. The design is function-based, not department-based, and is intended to enforce clearer traffic boundaries, easier ACL management, and lower lateral-movement risk.

Addressing Model

The office uses the private range 10.7.0.0/16 and breaks it into role-based /24 subnets for endpoints, internal servers, web-facing services, database/application data, guest Wi-Fi, cameras, VPN access, voice, general management, and out-of-band server management.

Configuration & Control Model

  • Inter-VLAN routing is controlled through the routed/firewall layer rather than unrestricted open routing.

Section 9

Cybersecurity, Data Protection & HIPAA Compliance

Security at Summit Care is layered into the design rather than added after the fact. The architecture combines segmentation, identity-backed wireless access, role-based controls, Microsoft MFA, logging, DLP, firewalling, secure administration, SNMP monitoring, and recovery-aware documentation practices.

Network Hardening

Internal, guest, public, database, management, and remote-access paths are separated. pfSense-style proxy filtering and Cisco firewall controls reduce exposure around edge and exposed services.

DLP & Data Handling

Email filters, file-transfer restrictions, and policy-aligned handling of regulated content help reduce accidental or unauthorized movement of patient and office data.

Monitoring & Alerts

Event Viewer review, SNMPv3 monitoring, scheduled PowerShell checks, firewall logs, and email alerts to IT or stakeholders create a more operationally useful security posture.

Secure Access Practices

  • WPA2-Enterprise wireless tied to Windows NPS/RADIUS with certificate-backed access where supported.
  • Organization-wide MFA across Microsoft-connected services and management portals.
  • SSH private key or certificate-based administration for Cisco and Linux systems.
  • VPN access through Cisco firewall VPN plus Linux-based backup VPN path.
  • Role-based access through AD groups, Entra visibility, and controlled app permissions.

SSH administration is treated as a controlled workflow rather than a convenience feature. Private keys or certificate-backed credentials are generated for approved administrators, stored in protected locations, referenced through the password manager and documentation process, and used only through known support paths. The Cisco ASA and Linux-based VPN model also supports this broader security design: the ASA remains the primary remote-access service at the edge, while the Linux-based software VPN provides a secondary path for continuity if primary access requires failover.

HIPAA-Aware Fit

Software and cloud-connected services were reviewed before adoption to confirm they better supported healthcare operations, privacy-aware administration, security visibility, and the regulated handling expectations relevant to Summit Care’s environment.

Section 10

Disaster Recovery & Backup Plan

Summit Care’s recovery design follows a practical 3-2-1 backup model supported by documentation, exported configurations, UPS-backed continuity, and rebuild baselines.

3-2-1 Recovery Layers

  • Primary on-prem copy retained on the local server environment
  • Secondary local copy retained on a Synology NAS with static IP and QuickConnect visibility
  • Offsite / cloud-connected copy retained through OneDrive and related managed storage paths

Backup Execution Model

PowerShell scripts and scheduled tasks run daily at approximately 11:50 PM to back up the server’s C drive, supporting system data, and the Practice Perfect database. This schedule is intended to reduce manual backup dependence and create a repeatable nightly protection cycle.

Recovery Readiness Measures

Recovery Control How It Supports Recovery
UPS Protection Protects the server during short power interruptions and unstable power events
Static Verizon Public IP Keeps remote access and predictable service connectivity more stable
Config Exports Cisco and Ubiquiti configs retained across server, NAS, and cloud locations
Documentation Copies Allows rebuild and validation even if one repository becomes unavailable
SNMP Alerts Email alerts notify IT and stakeholders when monitored thresholds or failures occur

Because DHCP continuity, baseline documentation, exported configurations, and image-based rebuild methods are all part of the design, the clinic can recover more predictably if systems fail, need replacement, or require clean reprovisioning from known-good standards.
